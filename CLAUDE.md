# anki-eng-training

Despite the directory name, this project does **not** use Anki or any flashcard app anymore (that direction was tried and explicitly rejected by the user as too much hassle). It's a plain-file English vocabulary habit: a static HTML reading page driven by a generated JS data file + CSV data sources.

## Theme

The user is reading *Mythos* by Stephen Fry (`data/story.pdf`) one chapter at a time and pulling vocabulary out of each chapter as they go — not freely chosen abstract themes anymore (that was the original direction; see "History" below for the archived version of that workflow).

There are no illustrations on the site. The book's images are licensed museum/Bridgeman/Alamy stock photos (confirmed via `pdfimages -list data/story.pdf`), and this site is public on GitHub Pages — re-publishing them would be redistributing copyrighted material without permission. This isn't a technical limitation, it's a deliberate call; don't re-litigate it without the user raising it again.

## Files

- `data/story.pdf` — the book (*Mythos*, Stephen Fry), 388 pages, 29 chapters across 4 "Parts". Source text only; never edited.
- `data/mythos-toc.csv` — static reference table of the whole book's structure. Columns: `order,part,chapter,page_start,page_end`. Generated once by scanning the PDF for heading lines; not regenerated by `render.py`. `order` is the chapter's position in reading sequence (1–29).
- `data/passages.csv` — source of truth for reading progress. Columns: `order,part,chapter,page_start,page_end,text,analysis_zh`. One row per chapter actually read, append-only, growing one row at a time as the user advances. `text` is the chapter's full original English text (not an excerpt). `analysis_zh` is a short (3–5 sentence) Chinese 賞析 commentary.
- `data/words-bank.csv` — vocabulary source of truth. Columns: `date,word,pos,ipa,en_def,zh_def,en_example,en_example_chunks,zh_example,source_chapter`. One row per word, append-only, every row tagged with the chapter title it was drawn from (e.g. `Out of Chaos`) via `source_chapter` — there's no untagged/general vocabulary anymore. (The ~90 pre-pivot abstract-theme words that predated `source_chapter` were deleted outright, not kept around unused — see "History" below.)
- `extract_chapter.py` — helper that finds the next unread chapter (by comparing `mythos-toc.csv` against what's already in `passages.csv`), pulls its text out of `data/story.pdf` via `pdftotext -layout` (requires poppler: `brew install poppler`), reflows the PDF's hard-wrapped print lines back into paragraphs (a new paragraph is marked by a 3+ space indent on its first line; continuation lines have none), and prints it for review. It does not pick vocabulary or write analysis — that's still a judgment call made by hand each time, same as word selection always has been.
- `render.py` — regenerates `book-data.js` by joining `words-bank.csv` and `passages.csv`. No third-party dependencies (stdlib only, `pdftotext` is only needed by `extract_chapter.py`).
- `book-data.js` — generated. A plain `const BOOK_DATA = {...}` data file: every chapter read so far, each with its own text/analysis/word list. There's no separate review/spaced-repetition sampling — every chapter stays browsable, so paging back to reread an earlier chapter's words **is** the review. Don't reintroduce a review-sampling feature unless the user asks; it was deliberately removed.
- `index.html` (formerly `vocab-today.html`, renamed for GitHub Pages so it serves at the repo root) — **static, hand-maintained, not generated.** Loads `book-data.js` via `<script src>` and renders it client-side (vanilla JS, DOM APIs — no `innerHTML` with interpolated text). Supports paging through every chapter read so far — prev/next buttons and a jump-to dropdown, current chapter tracked in `location.hash` (`#ch-N`) so it's bookmarkable and defaults to the latest chapter when absent. Each word/example has a 🔊 button that plays pronunciation via the browser's built-in Web Speech API (`en-GB` voice, no network calls, no API keys). It deliberately uses a `<script src>` tag rather than `fetch()` to load the data, because it also still needs to work when opened directly (`file://`, double-click, Chrome/Safari), where `fetch()` of local files is blocked by CORS but `<script src>` is not. A global 顯示中文/隱藏中文 toggle button (`#toggle-zh`) hides `.def-zh`/`.example-zh` by default (`body.hide-zh`) so the page can double as a self-quiz.
- `project-purpose.md` — short human-facing description of the repo.

Never hand-edit `book-data.js` — it's overwritten by `render.py`. `index.html` is the opposite: it's meant to be hand-edited (styling, layout, playback, navigation behavior) and is never touched by `render.py`.

### Sentence pause-grouping (斷句)

English example sentences are shown with `/` pause marks between reading chunks, so the user knows where to pause when reading aloud. This is **authored data, not computed code**: the `en_example_chunks` CSV column holds the same sentence as `en_example` but with `/` inserted at chunk boundaries (e.g. `"Marx argued / that the bourgeoisie had accumulated wealth / at the expense / of the industrial working class."`). `index.html` just splits that column on `/` and wraps each `/` in a styled `<span class="pause-mark">` — it does no linguistic analysis itself. The 🔊 play buttons still speak the plain `en_example` field (no slashes) so playback isn't affected.

A rule-based version (`chunkSentence()` doing punctuation + a fixed subordinator/conjunction word list) was tried first and lived in `index.html`, but rule-of-thumb chunking can't reliably handle prepositional-phrase grouping or "pause before the verb if the subject is long" — those need actual reading comprehension. Authoring the chunks by hand (or via an LLM) when each new word is added, using judgment guided by these four rules, produces far more natural breaks:

1. **標點符號**：遇到逗號（,）、句號（.）、分號（;）停頓。
2. **子句連接詞**：在 that, which, because, although 等連接詞前停頓。
3. **介系詞**：在 of, for, to, with 前停頓一下，把這組介系詞片語連在一起唸。
4. **長主詞**：如果句子主詞很長，可以在動詞出現前停頓。

These are guidance for judgment calls, not a mechanical algorithm — e.g. a short subject with an embedded prepositional phrase (`"Scholars of antiquity still debate..."`) usually reads better kept whole than broken at every rule-3 trigger.

## Reading workflow ("下一章")

The user drives this — they say "下一章" (next chapter) when they're ready, this isn't scheduled/automated. Progress is chapter-sequence-based, not date-based; multiple chapters can happen the same day or none for a while, doesn't matter.

1. Run `python3 extract_chapter.py` to get the next chapter's metadata (order/part/chapter/pages) and cleaned raw text.
2. Write a 3–5 sentence Chinese 賞析 (writing style, historical/cultural context, notable turns of phrase — see the "Out of Chaos" entry in `data/passages.csv` for tone).
3. Pick vocabulary from that chapter's own text — roughly 10 words as a floor (the user's English is weak, so err toward including more than fewer), more for longer chapters at your judgment. For each word write: word, part of speech, IPA, English definition, Chinese definition, an English example sentence **drawn from or closely adapted from the chapter's own text**, a pause-grouped version of that same sentence (`en_example_chunks`), Chinese translation of the example.
   - IPA must be **British (RP)**, not American: non-rhotic (no /r/ before a consonant or word-finally, e.g. `pɑːˈtɪʃən` not `pɑːrˈtɪʃən`), GOAT vowel `əʊ` not `oʊ` (e.g. `kəˈləʊniəlɪzəm` not `kəˈloʊniəlɪzəm`), LOT vowel `ɒ` not `ɑ` (e.g. `bəˈrɒk` not `bəˈrɑk`). When unsure of a specific word's British form, check a dictionary rather than guessing.
4. Append one row to `data/passages.csv` (order, part, chapter, page_start, page_end, full chapter text, analysis_zh) and the new word rows to `data/words-bank.csv` (with `source_chapter` set to the chapter title). Use Python's `csv` module (or equivalent careful quoting) when writing — several fields contain commas and the chapter text is multi-paragraph.
5. Run `python3 render.py`. This overwrites `book-data.js`; open (or refresh) `index.html` in a browser to see it — it should land on the new chapter by default (latest chapter).
6. Summarize what was added in the chat reply; don't just silently write files.

If the user asks to change the word-count floor, that's a judgment call for them — ask, don't assume.

## History: the old date-driven abstract-vocab workflow (superseded)

Before the book pivot, this project worked differently: each session the user asked for "today's words," ~10 history/humanities-themed words were picked freely (not tied to any text), appended to `words-bank.csv` keyed by that day's date, and `render.py --date` regenerated a `word-today.js` file (today's new words + a random review sample). That workflow, the `--date` flag, `word-today.js`, and the review-sampling feature are all gone — replaced entirely by the chapter-driven workflow above. Once chapters became permanently browsable, review sampling had no purpose, and the ~90 words collected under the old workflow had no `source_chapter` to attach them to any chapter view — so, per the user's call, that legacy data was deleted from `words-bank.csv` rather than left inert. If old date-keyed rows ever show up again in a diff, that's a regression, not intentional history to preserve.
